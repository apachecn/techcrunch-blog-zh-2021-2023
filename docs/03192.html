为了应对更严格的数据监管，企业应该寻求完全开放的 API

Jean-Paul Smets

想象一下这个场景作为一家年轻的企业:你是 Azure、AWS 或谷歌云平台的客户，假设它们是领先者。当您在俄罗斯或欧盟国家出差拓展业务时，您发现需要将数据存储在本地。更糟糕的是，在欧盟，你面临 GDPR，国家监管机构警告你在欧盟使用美国的云违反了 GDPR。

这是一个巨大的问题。无论您去美国以外的什么地方，您都必须遵守不同的法规，这些法规可能最终会阻止您成功部署应用程序。通过使用大玩家的 API，要么没有连接的可能性，甚至有法律风险。

企业如何解决这个问题？这就是基于开源软件的完全开放的 API 的巨大帮助和未来技术。

“完全开放”的 API 包含具有开放源代码操作程序的开放源代码软件，因此该技术可以在任何地区复制和审核，这解决了上述地缘政治冲突。

完全开放的 API 使最终用户能够控制如何调试软件(为 API 提供动力)，同时由于其可扩展性以及与闭环系统相比完全没有维护成本，还可能降低成本。这些封闭的系统受限于它们对特定平台的依赖性，这增加了开发人员的成本和其他限制。完全开放的 API 不只是利用开源软件，而是结合了对基础设施如何制造和如何运行的完整描述——这是一个完全开放的过程。

完全开放的 API 给任何开发人员在云中提供了与开源给软件行业带来的同等水平的控制和自由，而没有任何负面影响。

在欧洲，安全变得越来越重要。新的安全认证，如安全云计算的法德 ESCloud 标签，是旨在提高欧洲网络安全水平的合作范例。任何扩展美国法律域外适用的技术都可能很快被禁止进入政府市场和处理个人数据。

在许多国际市场，有关个人数据的当地法律要求所使用的服务或技术不含可能导致美国进行外国监控的技术成分。这造成了与立法密切相关的明显冲突，而提供 API 的美国公司处于其中。

采用完全开放的 API

要实现一个完全开放的 API，本质上需要一个开放的过程，让第三方独立于 API 的原始创建者来实现 API。为了使流程公开，所有步骤都需要以这样一种方式描述，即第三方可以再现它们并验证结果是否满足客户期望。

理想情况下，实现 API 的软件和硬件也应该是开源的。在无法审计源代码的情况下使用软件会带来后门存在的风险，这不符合某些数据保护法规。使用硬件而无法审核其设计会带来物流攻击的风险。对于他们自己的基础设施来说，这两种风险都很好理解，这些基础设施现在主要基于开源硬件和软件。
