AWS 通过 SnapStart 让 Lambda 冷启动延迟成为历史

在今晚的 re:Invent 动员会上，AWS 宣布了对其无服务器平台的一个小而重要的更新，解决了该服务最常见的问题之一。通常，当一个函数很长时间不使用时，Lambda 会关闭虚拟机——尽管有了更快的鞭炮微毫秒等改进，这仍然需要一段时间。现在，通过 SnapStart，AWS 通过创建客户 Lambda 函数的快照来解决这一问题，然后只需启动这些快照，而不必经过通常的初始化过程。

冷启动时间一直是 Lambda 最大的抱怨之一——然而正如 AWS 的公用计算高级副总裁 Peter DeSantis 在今天的主题演讲中指出的那样，峰值工作负载几乎是 Lambda(和所有其他无服务器平台)的初衷。凭借其鞭炮微管理器，AWS 已经将冷启动时间从几秒钟缩短到一秒钟以下。现在，该公司承诺通过使用鞭炮的快照功能，冷启动时间将提高 90%。

图片来源:弗雷德里克·拉迪诺瓦

这个新特性现在对所有 Lambda 用户都可用，尽管它必须对现有的 Lambda 函数启用，而且目前它只对使用 Corretto 运行时的 Java 函数有效。

一旦启用，当您第一次运行该函数时，它将执行标准的初始化。之后，它将创建内存和磁盘状态的加密快照，并缓存该快照以供重用。然后，当再次调用该函数时，Lambda 将获取缓存并启动该函数。缓存快照在 14 天不活动后会被删除。

正如 DeSantis 所指出的，像这样的改进将使更多的用户能够将他们的工作负载带到 Lambda 这样的平台上。他解释说，随着 Lambda 上鞭炮的推出，该公司已经看到了这一点。
