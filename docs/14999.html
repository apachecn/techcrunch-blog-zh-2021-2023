更低硬件成本的承诺刺激了初创公司将服务迁移到云，但许多团队不确定如何高效或经济地做到这一点。初创公司的开发人员认为他们可以维护多个独立于每个云提供商的应用程序代码库。

现在他们意识到管理起来太费时间了，而且试图满足每个人的所有需求并不光彩。

部署云基础设施还涉及分析工具和软件解决方案，如应用程序监控和活动日志记录，导致许多开发人员遭受分析瘫痪。这就是为什么云一夫一妻制是创业公司普遍接受的运营原则。但并不是每家公司都有在这些范围内无限期经营的奢侈。

实际上，在选择云基础设施提供商之前，分析可用的工具以控制应用程序成熟度和运行成本是非常重要的。

您需要:

  • 有经验的开发人员维护架构完整性、可维护性和许可考虑，或者
  • 为适应不断变化的环境以及构建、迁移和管理云应用而构建的云平台。

在你得到这些之前，这里有一些开始的最佳实践。让我们来看看初创公司在云方面面临的问题，如何定义云应用的结果，如何知道云基础设施何时需要更新，以及如何使用工具的组合。

分析你所处的位置，了解创业云的挑战

说到云基础设施，初创公司有两个级别:

在选择云基础设施提供商之前，必须分析可用的工具，以控制应用程序的成熟度和运行成本。
  1. 早期创业公司打造他们的第一个最小可行产品。这些公司希望部署最少的云计算来降低基础设施成本和技术决策，以便专注于产品和市场战略。
  2. 产品具有吸引力的创业公司。这些公司担心其云基础设施在安全性、可扩展性和可维护性方面的未来。然而，他们的规模不足以聘请专家团队。

这两个层次的创始人和决策者都在努力掌握管理云计算所需的专业技术。例如，一家中端市场初创公司找到我，他们已经在 AWS 中构建了自己的解决方案，但他们唯一的关注点是让它全部启动并运行起来(级别 1)。因此，它积累了技术债务，并且云架构很复杂，有数百台服务器、几十个独特的服务、第三方工具、部分日志记录和实现很差的服务网格。

然后，这家公司签下了一个中国的新客户，他坚持要在 Azure-China 上部署整个云解决方案，这是 Azure 的一个子集(2 级)。该公司在这种新环境中毫无头绪。

构建在不同云提供商上具有同等地位的并行解决方案可能成本高昂，并且需要付出巨大的努力。但是这家公司的另一个选择是失去一份重要的合同。他们别无选择。

为了复制和重新调整代码以在两个完全不同的环境中工作，该公司的开发人员在试图了解所有相关的实现、服务和考虑事项时可能会面临进一步的分析瘫痪。这就是为什么初创公司需要平台来创建云无关的架构、编写代码并自动部署到目标云，同时执行相关的测试和安全验证。

计算出你想要实现的结果

许多初创公司遵循云基础设施的“构建和修复模式”。这是因为初创公司的开发人员选择了他们看到的第一个工具，然后公司就被束缚住了(由于许可证或紧密耦合)。或者他们采纳某人的建议，就它如何与其他云层交互而言，这可能不是最优的。那么，缺乏对可用工具的适当分析和实验会导致尴尬的权衡和不良的业务阻塞。
