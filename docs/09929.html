远程办公的趋势在首席执行官和首席风险官之间制造了紧张，最终可能会做出并不总是符合企业最佳利益的决定。对于技术团队来说，这种脱节可能导致快速修复，最终让组织花费更多的钱，让个人付出更多的时间和压力。

远程革命将会持续。超过四分之三的工程师希望在家工作，一些国家正在考虑将远程工作纳入法律。在此之前，首席执行官和首席风险官需要回到同一页，以支持他们的技术团队和业务成果。

从 DORA 指标开始

DORA metrics 通过插入 Git(提交和拉取请求)和部署数据来测量软件交付速度和吞吐量。

通过查看 DORA 度量标准的组合，团队可以避免快速修复，以免恶化问题。

这些指标包括部署频率、变更的交付时间、平均恢复时间和变更失败率。这些信息还可以标记工程师是否正在失去参与度，突出表现优异和表现不佳的员工，并帮助公司避免求助于远程工作模式不可持续的快速解决方案。

无论您是准备越过混合中点，还是直接跳到远程，这就是您需要 DORA metrics 的原因。

照亮增长的领域

当在不同的地方工作时，流程更容易出错，沟通不畅，改进的机会也更容易被忽视。借助 DORA metrics，远程团队可以更有效地审查他们的实践并跟踪他们的成功。

为了有效地跟踪 DORA 指标，公司需要一个 CI/CD 平台(Jenkins、CircleCI 或 Github Actions ),该平台连接到一个开发分析平台，以汇总工程团队产出的数据。然后，团队可以放大度量标准，如变更的交付时间，以度量代码的质量，并最终度量他们维护的产品或服务的质量。

类似地，DORA metrics 可以加强远程团队在开发过程中包括测试的需求，以缩短他们的变更准备时间。这有助于消除瓶颈，因为测试人员可以教开发人员编写和自动化测试。
