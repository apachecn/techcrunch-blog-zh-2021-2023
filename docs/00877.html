在本世纪初，大多数关键业务软件都托管在私人运营的数据中心。但随着时间的推移，企业克服了疑虑，将关键应用程序转移到了云上。

DevOps 推动了这种向云的转变，因为它让决策者对托管在自己的数据中心之外的业务关键型应用程序有了一种控制感。

如今，企业正处于在其生产环境中尝试和接受机器学习(ML)的类似阶段，而这一变化背后的加速因素之一就是 MLOps。

与云原生创业公司类似，今天的许多创业公司都是 ML 原生的，并为他们的客户提供差异化的产品。但是绝大多数大中型企业要么只是现在才尝试 ML 应用程序，要么只是努力将功能模型投入生产。

以下是 MLOps 可以帮助应对的一些关键挑战:

很难让跨团队的 ML 协作发挥作用

ML 模型可以简单到预测客户流失，也可以复杂到决定圣何塞和旧金山之间的优步或 Lyft 定价。创建一个模型并使团队从中受益是一项极其复杂的工作。

MLOps 领域目前还处于早期阶段，但它具有巨大的潜力，因为它允许组织将人工智能引入生产环境，所需时间只是目前的一小部分。

除了需要大量标记的历史数据来训练这些模型之外，多个团队还需要协作来持续监控模型的性能退化。

ML 建模涉及三个核心角色，但每个角色都有不同的动机和激励:

数据工程师:训练有素的工程师擅长从多个来源收集数据，对其进行清理并以正确的格式存储，以便进行分析。数据工程师使用 ETL/ELT、数据仓库和数据湖等工具，并且精通处理静态和流数据集。由数据工程师创建的高级数据管道可能如下所示:

图片来源: Ashish Kakran，Thomvest Ventures

数据科学家: 这些专家可以在睡梦中进行复杂的回归分析。数据科学家使用 Python 语言、Jupyter Notebooks 和 Tensorflow 等常用工具，获取数据工程师提供的数据并对其进行分析，从而得出高度准确的模型。数据科学家喜欢尝试不同的算法，并比较这些模型的准确性，但在此之后，需要有人将模型投入生产。

人工智能工程师/DevOps 工程师:这些专家了解基础设施，能够将模型投入生产，如果出现问题，能够迅速发现问题并启动解决流程。

MLOps 使这三个关键角色能够持续协作，以交付成功的 AI 实施。

ML 工具的激增

在新的开发人员主导的、自下而上的世界中，团队可以从大量的工具中选择来解决他们的问题。
