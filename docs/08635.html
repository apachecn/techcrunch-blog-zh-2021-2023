抗议软件正在崛起:为什么开发人员正在破坏他们自己的代码

原文：https://web.archive.org/web/https://techcrunch.com/2022/07/27/protestware-code-sabotage/

Ax Sharma 是一名安全研究员和记者。他的兴趣领域包括开源软件安全、恶意软件分析、数据泄露和诈骗调查。

如果打击攻击和劫持 npm 等开源注册中心的合法软件还不够具有挑战性，那么应用程序制造商正越来越多地经历软件自我破坏的后果。开发者可以心血来潮，改变他们的想法，用他们的开源代码做任何他们想做的事情，反正大多数时候，是“原样”提供的，没有任何担保。或者，正如今年日益增长的趋势所见，开发人员故意破坏他们自己的软件库作为抗议的手段——将软件变成“抗议软件”。

7 月，被广泛使用的 atomicwrites Python 库的开发者 Markus Unterwaditzer】从流行的代码注册表 PyPI 中临时删除了他的代码，此前该网站表示将对“关键项目”的维护者强制进行双因素认证，这些项目在注册表中的下载量排名前 1%。Unterwaditzer 的 atomicwrites 项目符合标准，他的账户被要求注册双因素认证，他在一篇帖子中将描述为“为了保证少数依赖他的代码的公司(以我的空闲时间为代价)遵守 SOC2，一个令人讨厌且有资格的举动”。

一些人将这与 2016 年的left-pad 事件进行了比较，该事件在项目开发人员删除了他广泛使用的代码以示抗议后，短暂中断了互联网的很大一部分。开发者 Azer Koç ulu 与消息应用 kik 陷入商标纠纷，因为他的 npm 包被称为“Kik”在 npm 在争议中站在 Kik 一边后，kou Lu从 npm 注册表中撤回了他所有的代码——总共 273 个模块，包括广受欢迎的左键盘库。这完全是他力所能及的事，但这马上就产生了问题。当时，广受欢迎的 left-pad 软件包已经获得了超过 1500 万次的下载，甚至今天这个库还在继续被每周下载数百万次。因此，在 2016 年 3 月，当世界各地的开发人员因为他们的应用程序所依赖的左键盘组件再也找不到而导致他们的项目中断时，他们感到困惑和震惊。

几年前看似孤立的抗议在 2022 年死灰复燃，开发者破坏了他们自己的图书馆——有时是为了反对大公司，但最近是为了抗议俄罗斯入侵乌克兰。

抗议者最近的崛起

进入 2022 年的一周，数以千计依赖于大量使用的 npm 项目 colors 和 faker 的应用程序崩溃，并开始在用户屏幕上打印乱码文本。这不是一个恶意的演员劫持和改变这些合法的图书馆。原来这个项目的开发者马拉克·斯奎尔斯故意损坏了他自己的作品，以此向大公司发出抗议的信息。

Squires 的抗议是由 Log4j 项目维护者(大多数是开源志愿者)在 12 月假期修补关键漏洞的负担引起的。斯奎尔斯早些时候对财富 500 强公司免费使用他的开源代码而不提供财政支持或赞助维护表示失望。Log4Shell 漏洞只是强化了这种情绪——在应用程序中无处不在依赖 Log4j 的企业没有做足够的事情来支持那些在业余时间支持这些关键项目的无偿志愿者。

虽然 Squires 的抗议只是暂时冻结了依赖于 colors 库的项目，但几个月后，抗议软件的整个趋势随之而来，开发人员破坏了他们自己的项目，他们花了数百个小时来反对俄罗斯在乌克兰的战争。

2022 年 3 月，在俄罗斯军队进入乌克兰领土几周后，受欢迎的 npm 项目node-IPC——每周下载超过一百万次——开始清除涉嫌俄罗斯和白俄罗斯开发者的机器。该项目的开发者布兰登·诺崎·米勒据称破坏了代码，从而破坏了安装该代码的计算机。不用说， node-ipc 的被破坏版本——现在实际上是恶意软件——已经从 npm 注册表中删除。

从那以后，抗议软件的主题已经演变成开发商沉溺于更和平的抗议。像 event-source-polyfill 、 es5-ext 和 styled-components 这样的开源项目的新版本只是显示一条信息，敦促俄罗斯用户采取行动反对战争。因此，这些版本保留在 npm 上，因为它们不违反注册管理机构的政策。

发布抗议软件对开发者来说也不是一个容易的决定。它对被破坏的项目的所有版本进行额外的审查，这可能会伤害社区对开发者的信任。他们创作的任何软件，无论是过去的还是未来的，还能再被信任吗？

Evan Jacobs,styled-components 的主要维护者之一，告诉 TechCrunch，他的项目有积极主义的历史，“最显著的是我们对[黑人的命也是命]运动的支持，以及建议我们的用户考虑向平等正义倡议捐款。”他补充说:“我听说俄罗斯政府开始审查西方新闻网站，并意识到我们有一个独特的机会，通过一个非典型的渠道传递一个简洁、信息丰富的信息:我们的 npm 包安装。”

npm 上 nestjs-pino 项目的截图，突出显示了一张儿童在乌克兰马里乌波尔防空洞等待的照片。图片来源: TechCrunch /截图

雅各布斯认为，俄罗斯人获得不受国家干预的关于战争的准确消息至关重要。他修改了 styled-components ，截至 4 月份，其月下载量超过 1500 万次，向俄罗斯用户显示双语信息，总结“俄罗斯军队在乌克兰犯下的许多暴行”

“它产生影响了吗？我们可能永远不会知道，”雅各布斯说。“也就是说，我认为这是完全值得的，有机会传播信息，并希望抓住俄罗斯软件人员的眼睛，否则他们可能不会看到正在发生的事情。”

另一个开发者，Mariusz Nowak，es5-ext 项目的创建者，修改了该库的最新版本，将俄罗斯和白俄罗斯的用户引向准确的新闻来源，如 BBC 的 tor 服务。诺瓦克告诉 TechCrunch 修改代码的决定，称这是因为俄罗斯人“不太清楚发生了什么，他们受到了宣传媒体的影响”，指的是国家对俄罗斯媒体的严格控制。诺瓦克说:“只有当你在俄罗斯安装软件时，这条信息才会显示出来，在世界其他地方是看不到的。”。

诺瓦克说，使用他的开源库进行活动并没有影响他在更广泛的社区中的可信度，但他在开始时确实收到了一些愤怒的回应。

雅各布斯和诺瓦克并不是唯一重组他们的开源代码来抗议战争的人。软件供应链安全初创公司 Socket 告诉 TechCrunch， nestjs-pino 是一个受欢迎的 npm 项目，每周下载量超过 100，000 次，它更新了其主要的“readme”文件，以引导人们关注乌克兰正在发生的危机。与软件包捆绑在一起的安装脚本也会在安装完成后立即打印出一条控制台消息。

“你不能相信你无法验证的东西”

开源开发者正在发现新的和创造性的途径，不再限制他们为他们的项目实现新的特性，而是通过为某个原因修改他们的项目来积极表达他们对更大的社会问题的观点。此外，与必须按照付费客户的期望运行的专有代码不同，大多数开源许可证都非常宽松——对消费者和开发人员都是如此——提供代码的许可证不能保证开发人员不应该也永远不会对他们的代码做什么，这使得抗议者成为捍卫者的灰色地带。

事实上，作为 Sonatype 的一名安全研究员，我观察到抗议者如何在早期阶段对我们构成挑战，以及我们如何调整我们的自动化恶意软件检测算法，以便现在通过像 colors 和 faker 这样的项目捕捉自我破坏。传统上，该系统旨在发现上传到开源库的域名仿冒恶意软件，但像恶意劫持或开发者在没有警告的情况下修改自己的库这样的情况需要更深入地了解抗议者如何工作的复杂性。

这一主题也让像 npm 这样的主要开源注册中心——由微软子公司 GitHub 所有——在处理这些边缘案例时处于十字路口。

Socket 的创始人费罗斯·阿巴克哈迪贾告诉 TechCrunch，像 GitHub 这样的注册中心处境艰难。“一方面，他们希望支持维护者的言论自由权，以及利用他们的平台支持他们所信仰的事业的能力。但另一方面，GitHub 对 npm 用户有责任确保恶意代码不会来自 npm 服务器。这有时是一个困难的平衡，”Aboukhadijeh 说。

确保您在构建中只获得组件的审查版本的一个简单解决方案是固定您的 npm 依赖版本。这样，即使项目的未来版本被破坏或劫持，您的构建也将继续使用“固定的”版本，而不是获取最新的、被感染的版本。但这并不总是对所有生态系统都有效的策略，比如 PyPI，其中组件的现有版本可以重新发布——正如我们在劫持 ctx PyPI 项目 的案例中所看到的。

“围绕‘抗议软件’的对话实际上是关于软件供应链安全的对话。专注于软件供应链安全的初创公司 Chainguard 的联合创始人兼首席执行官丹·洛伦茨(Dan Lorenc)对 TechCrunch 表示:“你不能相信你无法证实的事情。

Lorenc 反对阻止抗议者的建议是遵循良好的开源安全卫生和最佳实践，这可以帮助开发人员更容易和更早地开发抗议者。“了解和理解您的依赖性，对您在环境中使用的开源代码进行定期扫描和审计是一个开始。”

但 Lorenc 警告说，关于抗议者的辩论可能会吸引模仿者，他们会加剧问题，并分散开源软件捍卫者的注意力，使他们无法专注于解决真正重要的问题——阻止恶意行为者。有了抗议者，还有未知的未知。对抗议者来说，什么问题太小或太大？

虽然实际上没有人能够规定开源开发者可以用他们的代码做什么——这是开发者一直拥有的权力，但现在才刚刚开始利用。
