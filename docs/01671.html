Adtech 被发现通过 IAB TCF 跟踪拒绝同意的欧盟用户

原文：https://web.archive.org/web/https://techcrunch.com/2021/12/17/adalytics-tcf-research/

新的研究调查欧洲互联网用户登陆一个广告支持的网站并表达他们的“隐私选择”后会发生什么——使用一个旗舰广告业同意管理平台，该平台应该允许他们控制他们接收的广告类型(即非跟踪与“个性化”)——对 IAB 欧洲自封的透明度和同意框架(TCF)提出了新的问题。

TCF 已经陷入隐私监管机构的困境。

上个月，IAB 欧洲宣布，它预计将被发现违反了欧盟的一般数据保护条例(GDPR)——该框架也将被发现违反。尽管 IAB 试图表明，一些调整将足以解决比利时数据保护局(DPA)发现的问题。

我们仍在等待比利时当局最终决定的公布。但是它的初步发现——去年——凸显了一系列 GDPR 的失败。

尽管如此，IAB 继续辩称，TCF 正在为据信参与该系统的近 800 家广告技术供应商提供服务，并大声驳斥了对它的批评。例如，其首席执行官 Townsend Feehan 在本月早些时候对批评嗤之以鼻——告诉 Engadget “如果用户说不，这些(追踪)就不会发生”。

然而，这项新的研究对以下说法提出了质疑:如果用户通过 IAB 的 TCF 对跟踪/行为广告说“不”,广告技术行业就会乖乖就范。

这项研究的一个关键部分是研究广告技术生态系统如何响应只请求基本广告的用户信号，即不基于跟踪的广告，以研究当用户拒绝“个性化”广告时广告供应商如何响应。

在这里，研究人员发现有证据表明，当许多广告技术供应商明确表示他们不想要基于跟踪的广告时，他们仍在继续跟踪和描述互联网用户。

虽然许多早期研究发现欧盟出版商在实施 cookie 许可方面存在问题，例如在请求网站访问者许可之前跟踪被丢弃的 cookie，但这项由 adtech 研究员 Adalytics 开展的新研究旨在关注 TCF 框架本身——通过检查广告支持网站忠实报告用户广告选择的情况。

此后有问题的数据流牵涉到广告技术行业本身——以及它对 TCF 作为旗舰合规工具的声称——因为它表明，一旦用户的“隐私选择”被传递给广告“合作伙伴”，该框架未能准确反映和实际上尊重用户的“隐私选择”。

Adalytics 列举了一系列要点，写道这些发现表明:

  1. 许多主要的广告技术供应商继续跟踪和分析欧盟用户，即使有欧盟用户明确反对
  2. 由 IAB Europe 设计的 TCF 字符串似乎没有被许多广告技术供应商认可或正确解析
  3. 一些广告技术供应商可能无法证明他们是在用户同意的情况下获得用户数据的，这可能会使他们面临合同合规性、投资者/股东披露和监管风险
  4. 在许多情况下，用户不可能通过允许“基本广告”来支持媒体创作者，同时不允许跟踪和行为剖析来保护他们自己的隐私

尽管值得注意的是，研究人员通过 Chrome 开发者工具能够观察到的东西是有限的——考虑到在广告技术公司自己的服务器上进行的任何处理都无法通过这种外部研究进行验证。

一旦 adtech 生态系统得到了人们的数据，就很难理解人们对这些数据做了什么。但这也触及了基于监控的广告在遵守 GDPR 方面的问题的核心——该法案还要求在处理人们的数据时要有问责制、透明度和安全性，以及这样做的有效法律基础。

即使抛开这些重大问题不谈，当一个人明确表示不应该这样做时，跟踪被丢弃的 cookies 和被传递的用户数据这一事实，对 IAB 的 TCF 来说，看起来很尴尬。

为了研究框架的 adtech 端的数据活动，研究人员在欧盟的一些国家进行了测试——访问他们手动验证了正确配置框架以发送用户同意字符串的网站，并只选择基本广告；拒绝个性化/基于跟踪的广告/档案等，并将 adtech 处理器的选择限制在单一供应商。

测试人员还确保反对“合法利益”,这样他们的同意偏好就不会以这种方式被绕过。

如果 TCF 像费恩本月早些时候向 Engadget 发表的言论所暗示的那样运行——也就是说，如果用户可以通过 TCF 简单地说“不”来否认跟踪——那么研究人员本应该只观察到个人指定应该在流动的数据。

相反，他们发现——在大多数情况下——数据流看起来与表达的选择非常不同。

这篇论文还详细描述了许多追踪 cookies 在用户同意选择之前被设置的例子。(尽管他们说他们从分析中排除了这样的例子，因为他们的目标是研究用户通过 TCF 提交选择后会发生什么。)

在对 adtech 供应商的研究中引用的例子似乎忽略了拒绝跟踪的 TCF 信号，包括一个拥有比利时 IP 地址的访问者访问一个受欢迎的当地新闻网站，nieuwsblad . be—，他只同意基本广告，只同意来自谷歌的广告(所以他们明确拒绝“个性化”广告和配置文件)——然而，在检查网络 HTTP 请求的 Chrome 开发工具时， 观察发送到 AppNexus(又名 Xandr)拥有的ib.adnxs.com的 域的 HTTPS 请求，该请求通过丢弃一个名为“uuid2”的 cookie 进行响应，该 cookie 设置为三个月。

“鉴于[该用户]反对个性化广告并创建个性化个人资料，只向谷歌提供了同意，而且这些同意选择直接包含在向 adnxs.com 发出的 HTTP 请求中，尚不清楚为什么 AppNexus 服务器会通过在[该用户的]浏览器 中设置持久的广告相关 cookie 来做出回应，”研究人员观察道。

在另一个例子中，一个拥有法国 IP 地址的用户访问新闻网站 lemonde.fr，尽管不同意同意横幅中提供的任何 cookie 或目的，但他们再次看到 HTTPS 请求被发送到 id5-sync.com，该请求通过设置一个名为“id5”的 cookie 三个月来响应，并触发 302 HTTP 重定向以与 rtb-csync.smartadserver.com 同步数据。

“这个发送到 id5-sync.com 的特定 HTTPS 请求在一个名为“gdpr _ consent”的查询字符串参数中包含了前面提到的 TCF 字符串，”研究人员报告说，并补充说:“域名id5-sync.com属于 ID5，这是一个总部位于伦敦的数字广告行业的"身份平台。”

他们进一步指出，在 github 的概述中，ID5 通用 ID 被描述为“出版商、广告商和广告技术平台可以用来识别用户的共享、中立的标识符”。

所以，再一次，如果用户说他们不想被识别和跟踪广告，为什么 id5 cookie 会被丢弃呢？

在研究中详述的另一个例子中，也涉及一个法国 IP 地址——这次用户访问报纸latribune . fr——用户的同意选择显然再次被扔进虚拟垃圾箱。

在这种情况下，用户已经指定他们想要由美国供应方平台 OpenX 提供的基本广告。

然而被观察到触发用户 ID 与“众多其他方”同步，包括亚马逊、谷歌、DataXu、AppNexus、蜂蜡(bidr . io)、阿迪斯预测数据平台( 、ipredictive.com)、AdPilot(【erne.co)、Simplifi 控股()

在另一个例子中——涉及一个法国 IP 用户访问Atlantic o . fr；仅同意来自谷歌的基本广告——用户看到“许多 HTTPS 请求都是用这个 TCF 同意字符串发出的，其中一些似乎是用户数据同步或设置跟踪 cookies”。

研究人员继续注意到: “一个请求发送到s . cpx . to响应通过设置一个 1 年的 cookie 叫做“cpSess”。该域名由总部位于伦敦的 Captify 所有，而“cpSess”cookie似乎被用于 存储和链接用户的个人信息”——然后引用另一个消息来源称，该 cookie“被用作[…]广告公司的跟踪机制”，并且“有助于在用户浏览时发送有针对性的营销广告”。

这项研究详细列举了许多其他在用户要求不被跟踪后被观察到的意外数据流和同步的例子。

研究人员还详细介绍了大规模自动化测试的结果，以及上面引用的手动示例——基于来自 48，698 个不同出版商域的爬虫数据——在这里，他们发现了“成千上万”广告请求的证据，这些请求错误地声称 GDPR 不适用于实际上在欧盟的网络用户…

在 35，389 个发布者中，发现有一个 HTTPS 请求被发送到广告技术供应商，并且包含“gdpr=0”查询字符串参数或“gdpr=1”查询字符串参数，其中 28，941 个(81.8%)包含至少一个发送到广告技术供应商的 HTTPS 请求，其中查询字符串宏参数“gdpr=”被设置为“0”，这意味着他们在发信号通知 gdpr 不适用。

更多来自 Adalytics 的文章:

“根据 IAB TCF 文档和谷歌文档，1 表示 GDPR 适用，0 表示不适用。许多广告技术供应商被观察到在“gdpr=0”时处理德国和芬兰的用户数据，尽管爬虫没有点击 CMP 同意横幅。在每一种情况下，广告技术供应商都可以通过执行简单的 IP 地址地理位置查找来验证用户确实在欧盟。然而，似乎许多人在发送跟踪 cookies 或执行用户数据同步之前，没有执行任何额外的服务器端检查，就从表面上接受了这个“gdpr=0”参数。

为了验证这一发现，Adalytics 表示，它联系了一位程序化媒体买家，该买家向欧盟、英国和瑞士的用户投放 广告。这位买家的广告总共被派发了 1.11 亿次，而“gdpr _ applies”宏在 42.3%的时间里被设置为“0”，“这表明所涉及的广告技术公司声称 gdpr 不适用于这些用户”。

然而，在使用一种 IP 地址地理位置查找服务 检查这 4700 万条广告的投放地点时，Adalytics 报告称，广告客户发现大部分广告投放给了西班牙、克罗地亚、意大利、法国和其他欧盟国家的用户，这意味着 GDPR 确实适用……

“如果广告技术公司不通过执行 IP 地址地理位置检查来检查“gdpr _ applies”字符串的完整性，他们最终可能会在适用于他们广告活动的相关法规方面误导他们的广告客户，”Adalytics 继续警告说。

调查结果进一步强调了广告欺诈问题的严重性——而且，就 IAB 欧洲的 TCF 框架而言，表明它仅仅提供了一个合法合规的外表，因为它实际上没有能力做广告上声称的事情。

根据研究结果，广告技术供应商经常没有正确实施和部署它——使“透明度和同意框架”充其量只是一个骗局；或者，嗯，合规剧场。

难怪整个系统都处于监管困境中——尽管欧盟监管机构花了很长时间来询问广告技术行业索赔背后的现实，这仍然让任何关心民主监督和基本权利的人感到极其不安。

“我们在研究中得出的结论是，广告技术供应商没有花费时间、金钱和工程工时来正确配置他们的服务器和 API，”Adalytics 的创始人 Krzysztof Franaszek 在讨论这项研究时告诉 TechCrunch。“理论上，所有 TCF 参与者……都应该以一种检查和验证 TCF 字符串的方式设置他们的服务器和 API。

如果 API 接收到一些用户数据，而 TCF 字符串显示用户并未同意这些数据，则供应商应立即丢弃这些数据，并避免对其进行任何操作

Franaszek 指出，研究人员确实观察到了至少一个正确行为的例子 AppNexus 收到了一个不允许个性化的 TCF 字符串，或者一个名为 Index Exchange 的供应商获得了数据，并“正确/正当地配置了他们的服务器，以 HTTP 400 错误代码做出响应，这表明‘由于隐私信号，请求失败’”。

然而，他指出，这正是“所有 795 家 TCF 公司应该做的”，而研究显示——恰恰相反——许多公司根本没有检查弦。

爱尔兰公民自由委员会的高级研究员 Johnny Ryan 也对调查结果进行了评论，他是一名长期的广告技术举报者，曾对实时竞价(RTB)滥用个人数据提出过无数投诉，他说这项研究只是“TCF 只是一个坏主意”的最新例证。

“没有控制。Ryan 告诉 TechCrunch 说:“没有任何技术措施来控制数据的变化。“这意味着 TCF 永远无法实现其声称的提供控制和透明度的目标。

“我们向比利时 APD [DPA]提出的核心论点是，由于 RTB 固有的和众所周知的不安全，TCF 不可避免的失败意味着，除了成为所有欧洲人生活中不必要的麻烦之外，TCF 还违反了[GDPR]的数据最小化原则，因为它出于不可能实现的目的处理 TCF 字符串个人数据。”

瑞安还指出了一些 2017 年 IAB 的信件——这些信件在 2019 年通过 FOI曝光——费恩当时正在寻求就电子隐私规则游说欧盟委员会，暗示同意与 RTB 不相容——在她给欧盟立法者的电子邮件中附上一份文件，指出“从技术上讲，用户不可能事先了解实时竞价(RTB)场景中涉及的每个数据控制者的信息”。

在 GDPR 法律下，同意必须是具体的，知情的，而且是自愿的。

正如 Franaszek 所说，Adalytics 的研究有一些局限性，例如人工部分仅限于少数“似乎正确编码用户同意输入的出版商，以排除出版商或 CMP 的错误导致未经用户同意进行用户建档的可能性”。

同样不清楚的是，在使用 TCF 系统的 795 家广告技术公司中，有多少/多大比例没有恰当地尊重用户的选择。上面引用的用户拒绝同意被正确响应的例子表明这是可能发生的。不过，这种情况发生的频率完全是另一回事。

无论如何，研究人员观察到的那些显然不尊重用户否认跟踪的供应商包括“几家价值数十亿美元的跨国公司，如 Rubicon、Pubmatic、亚马逊、谷歌和 Trade Desk”，per Franaszek 说。

“我认为，如果你看看我们调查的供应商的股票市值，它远远超过数字广告行业的 50%，”他补充说。

一个更普遍的警告是，鉴于广告技术行业高度不透明的特点，总是很难理解数据是如何处理的——研究人员承认，在论文讨论局限性的一节中，解释数据需要很多“细微差别”。

这项研究也没有经过正式的同行评议，所以如果发表在同行评议期刊上，也没有经过严格的学术程序。

但是，好吧，这些发现对 adtech 来说肯定不是什么好事。

根据 IAB TCF 框架 1.0 版本的 github 知识库上的公开文档，广告行业机构建议，他们当时即将推出的“技术行业解决方案”将允许网站运营商:

  1. 控制他们希望允许访问其用户浏览器(用于设置和读取 cookies)并处理其个人数据的供应商，并向在线广告生态系统中的其他方披露这些选择
  2. 根据 ePrivacy 指令(用于设置 cookies 或访问设备上信息的类似技术应用程序)和/或符合适用法律要求的 GDPR 寻求用户同意，并通过在线广告生态系统表明同意状态

从这些陈述的能力中自然引出的问题是，然后呢？当用户的“隐私选择”被告知吞噬数据的广告技术生态系统后，会发生什么？TCF 实际上如何确保广告商尊重人们的选择？

Adalytics 的研究只是表明行业标准弹出“隐私选择”是海市蜃楼的最新研究；一个暗示性的选择幻觉被包裹在一个同意的哑剧中，试图打破监视广告的不可能循环:GDPR 顺从。

因为这似乎是确保你要求 TCF 传递给广告技术供应商的“选择”得到尊重的唯一方法，就是你要求继续跟踪…

我们联系了 IAB，请其对 Adalytics 的研究发表评论，并询问 IAB 是否采取任何措施来验证参与的广告供应商是否遵守表达用户选择的 TCF 字符串。

我们还询问 IAB，鉴于比利时的违约调查结果，它打算如何调整该框架——以及它自己的活动——以便符合 GDPR 的要求。

我们将根据任何回复更新此报告。

更新:欧洲 IAB 现已做出回应——发送以下声明:Adalytics 的研究:

“当正确实施时，TCF 允许互联网用户向网站及其供应商传达他们不希望其数据被收集或处理的信息。作为 TCF 参与者的所有供应商都应遵守其在 TCF 下的承诺，包括在用户拒绝或反对时不处理任何数据。

“TCF 主要是公司可以在其商业关系中实施的政策和技术标准，但是使用 TCF 并不能减轻任何供应商完全遵守 GDPR 的责任。虽然确保正确应用 TCF 是供应商的责任，但 IAB 欧洲公司提供支持以确保正确实施 TCF 并解决不合规问题。

“9 月，我们启动了 TCF 供应商合规计划，该计划使我们能够监控供应商合规情况，并强制执行违规行为。这包括禁止在未经用户同意的情况下设置非必要的 cookie 或在没有法律依据的情况下处理用户的个人数据，验证供应商使用 cookie 的期限，并确保用户的选择被如实转发。"

对于比利时德新社对 TCF 的调查，该社表示，“对于正在进行的调查”，它不能进一步置评。

“一旦裁决最终确定，我们将与相关当局合作，解决向 TCF 提出的任何问题，”IAB 补充道。
