设计一个好的 MVP 意味着跳出框框思考

原文：https://web.archive.org/web/https://techcrunch.com/2022/05/03/your-mvp-is-not/amp/

图片来源: 哈吉坎普斯 / 哈吉坎普斯为 TechCrunch 制作(图片已修改)

每当我与产品驱动的初创公司创始人谈论最小可行产品时，我经常发现自己陷入令人沮丧的谈话中。MVP 这个术语是一个严重的用词不当；好的 MVP 是没有生命力的，当然也不是产品。仔细想想，它可能也不像你希望的那么小。

在精益创业的世界里，创始人必须保持高度专注于找出如何尽快失败。理想情况下，你失败了，这意味着你最终拥有了一家正常运转的企业。很多“尝试失败”的方法都涉及到寻找你的商业机会，思考你的企业将来可能会失败在哪里。那就去搞清楚那部分。

如果整个客户群已经很乐意使用易贝，并且不会离开，即使你的产品很好，那么建立世界上最好的出售豆豆娃的平台也没有好处。如果滑板车公司不关心滑板车是否被盗，那么专门为骑行共享滑板车制造一把大锁是没有好处的。如果在你写一行代码之前就有办法知道是否有人会买你的产品，那就太好了。

那么 MVP 从何而来呢？作为创业公司，你有一个假设；MVP 是你为证实或消除你的假设所能做的最少的工作。埃里克·里斯(Eric Ries)——没错，就是写了《精益创业》(The Lean Startup)的那个人——以 Dropbox 的 MVP 为例。这不是一个功能齐全的成熟产品。它并不是一个去掉了许多功能的产品。这是一个视频，展示了一个产品是如何工作的。对该视频的回应是该公司需要的确认:如果他们建造了它，他们将能够为其尚未建造的产品找到客户群。这就是他们所做的:制造产品，并获得巨大成功。

设计一个好的 MVP

设计一个好的 MVP 意味着跳出框框思考。你能写多少代码？你可以不做设计吗？如果你最大的问题是，你能否以合理的客户获取成本吸引客户，你能否只做一个广告活动和一个结账页面，然后只给下订单的人退款？如果这听起来很有趣，但你担心品牌风险，你能创造一个假品牌，并得到你的产品的答案吗？

诀窍是仔细思考假设——关于你的产品、市场、你正在进入的问题空间、你希望吸引的客户以及竞争格局，什么是真实的？你有多大把握你的假设是正确的？设计一个好的 MVP 是一门艺术，但它始于一个真正好的问题。这里有几个例子:

  • 我们有没有可能将四个小时的手工会计任务缩减为一个三分钟就能运行的脚本？这是一个技术 MVP——你可能需要拼凑一些代码，看看你是否能可靠地自动化手动任务。
  • 我们能找到愿意付费来自动完成这项任务的人吗？在某些情况下，答案是“不”——是的，你可能会为初级会计师节省一些时间，但在某些行业，人们根本不在乎初级职员花多少时间做手工。在这种情况下，你需要确定你是否能找到 20-30 个愿意为此付费的客户。记住，有人说“哦，听起来是个好主意”和他们把手伸进口袋实际上付给你钱是不同的。
  • 设计对该产品重要吗？许多 B2B 软件丑得吓人。这不是因为优秀的设计师不存在，而是因为这根本不是首要任务；必须使用产品的人可能更喜欢更好的设计或更简单的 UX，但决策者不在乎，用户也没有发言权。换句话说:如果你找不到一个商业案例，就不要把一半的开发预算花在更容易使用的东西上。尤其是当你在这个过程中无意中开发了错误的特性集。
  • 现任会复制我们并摧毁我们吗？如果你所在的领域有很多现有公司，做一些研究，看看他们对其他初创公司的反应如何。如果他们倾向于获得它们，很好。如果他们倾向于复制他们的功能和创新，然后摧毁它们，那就没那么好了。稍微谷歌一下(当然，阅读你所在行业的 TechCrunch)可以让你在未来省去很多头疼的事情。如果现任者经常窃取创新，那就在专利上多投资，并为律师留出一些钱。
  • 这个功能对我们的客户有意义吗？也许你的少数客户会大声询问相同的功能，但你不会是第一个大张旗鼓地推出新功能却遭到集体耸肩的公司。大声喧哗的客户并不代表你的整个客户群，所以在如何整理你的积压订单时要明智——如果一个功能没有为你公司的整体业务目标增加重要价值，不要优先考虑它。围绕这一点设计 MVP 的一个方法是在你的 UI 中添加一个按钮，并跟踪有多少人点击了它。抛出一句“快到了！”消息，例如。是的，这对用户来说很烦人，但这比花费几个开发周期来添加一个几乎没人会使用的特性要“便宜”得多。

简而言之，关键是要非常仔细地思考问题是什么，然后想出优雅、低调的方式来问这个问题。调查能代替航运法规吗？一个视频演示能给你想要的答案吗？你能打电话给 50 个客户，问他们谨慎的问题，看看他们是否建议你正在考虑的功能作为问题的潜在解决方案？他们可能会以两种方式让你吃惊:你的客户可能会非常想要你建议的东西(太好了！)，他们可能讨厌它(这也很好，这意味着你不必浪费时间和金钱来开发他们不想要的东西)，或者他们可能有一种完全不同的解决问题的方法，这种方法恰到好处，开发成本更低，并有助于他们感受到参与到你的过程中。

我没有给 MVP 一个更好的名字的建议，只是不要陷入把它想成一个产品的陷阱，是可行的，或者是必要的，是小的，简单的或容易的。有些 MVP 很复杂。这个想法是，尽可能少的花费你宝贵的资源来得到你问题的答案。
